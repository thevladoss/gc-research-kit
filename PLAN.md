# GCHAR — план после калибровки

Калибровка (главы 04–06) одобрена пользователем 2026-07-05. Этот файл — план оставшихся фаз
для любой будущей сессии. Текущее состояние всегда в `manifest.json`; правила пайплайна — в
`CLAUDE.md`; запуск одной главы — команда `/process-chapter N`.

## Фаза 1 — батч-обработка оставшихся глав

Охват: главы 00–03 и 07–42 (40 глав), каждая через все 4 стадии `/process-chapter`.

Порядок работы:

- **Батчи по 5 глав параллельно.** Разные главы можно вести одновременно (калибровка
  одобрена), но внутри одной главы стадии строго последовательны.
- **Коммит после каждой главы**, прошедшей стадию 4 (по образцу коммитов глав 04–06).
- **Отчётность:** одна строка сводки на главу (число утверждений, вердикты, подтверждённые
  расхождения, load-bearing, число поисков) + дайджест после каждого батча из 5 глав.
- **Контроль бюджета поиска:** ожидаемая норма 12–20 поисков на главу (verify + defend).
  Если глава выходит за 20 — отметить в дайджесте и назвать причину (не молчать и не
  останавливать батч из-за одиночного выброса).
- **Дельта-формат:** verifier пишет дельту на главах длиннее ~8k слов (07, 08, 12, 18);
  defender и impact-assessor — на главах с >120 утверждений. Оркестратор сливает дельты
  через `scripts/merge_stage.py` и после КАЖДОГО merge гоняет `scripts/validate.py`
  (подробности в `.claude/commands/process-chapter.md`).
- **Schema-гейт:** невалидный JSON = переделка стадии её же агентом (SendMessage/повторный
  запуск), никаких ручных правок данных.
- После каждой главы обновлять `manifest.json` (`scripts/update_manifest.py`).

## Фаза 2 — синтез

Условие входа: ВСЕ 43 главы (00–42) имеют `impact: done` в manifest.

- Один запуск сабагента `synthesizer` (модель `claude-fable-5`, задана в frontmatter).
- Вход: все `impact/*.json`. Выход: `synthesis/aggregate.json` + русскоязычные
  `synthesis/clusters.md` и `synthesis/central_thesis_analysis.md`.
- После завершения: `scripts/update_manifest.py --global synthesis done`, коммит.

## Фаза 3 — итоговый отчёт

> **Статус (2026-07-05): вытеснена Фазой 2c.** Отчёт `report/index.html` собран по данным
> Фазы 1 и больше не обновляется: его заменит сайт Фазы 2c, который строится поверх
> адъюдикационного слоя (Фазы 2a/2b). Файл остаётся в репозитории как артефакт Фазы 1;
> править его не нужно. Содержательные требования ниже переходят к сайту Фазы 2c.

`report/index.html` — единый самодостаточный интерактивный отчёт на русском. Все данные
инлайнятся в файл (никаких внешних запросов), собирает основная сессия.

Обязательное содержимое:

1. **Тепловая карта глав** по плотности расхождений и impact_score.
2. **Фильтруемая таблица утверждений**: фильтры по вердикту, типу, severity, narrative_arc;
   в строках цитаты из GC и ссылки на источники.
3. **Визуализация цепи центрального тезиса** (apostolic purity → … → end-time crisis)
   с маркерами повреждений на load-bearing звеньях.
4. **Метрика доли anachronistic** среди обвинительных вердиктов — на видном месте
   (ключевой показатель «ошибка эпохи vs авторская выдумка»).
5. **Выводы синтеза** из фазы 2.

Требования к исполнению: чистый читаемый интерфейс, тёмная тема поддержана, весь UI на
русском (цитаты остаются на английском с русским комментарием). После сборки:
`scripts/update_manifest.py --global report done`, финальный коммит.

## Фаза 2a — адъюдикация 97 disputed-вердиктов

Фазы 1–3 завершены. Фаза 2a добавляет слой градуированных решений поверх 97 утверждений
с итоговым вердиктом `disputed`. Правила заморожены в `ADJUDICATION_CHARTER.md`
v1.0-frozen; разбиение на 23 досье — в `ADJUDICATION_DOSSIERS.md`; состояние —
секция `adjudication` в `manifest.json`. Запуск одного досье: `/adjudicate DNN`
(свежий контекст сабагента `adjudicator` на каждое досье, хартия §10.6).

Порядок работы:

- **Калибровочные ворота (хартия §10.5).** Первым идёт ТОЛЬКО D01 (chronology-1844,
  15 из 16 утверждений load-bearing, повышенный бюджет 18–20 поисков). После него СТОП:
  результаты пользователю на разбор. Дальнейшие досье не открываются, пока пользователь
  не одобрит калибровку; после одобрения выставить
  `adjudication.calibration_approved = true` в manifest (отдельный коммит).
- **После одобрения — батчи по 3–4 досье** в порядке `ADJUDICATION_DOSSIERS.md`
  (D02 → D23). Досье внутри батча независимы (разные доказательные базы) и могут идти
  параллельно; одно досье никогда не делится между контекстами и не пересекает батчи.
- **Schema-гейт:** после каждого досье `python3 scripts/validate_adjudication.py
  adjudication/DNN_<slug>.ruling.json`. Невалидный файл → один повторный запуск агента
  с текстом ошибок; вторая неудача → стоп и отчёт. Ручных правок ruling-файлов нет.
- **Коммит после каждого досье** (ruling-файл + manifest), сводка в чат по-русски:
  распределение решений как написано / по ослабленным версиям, framing gaps,
  провалившиеся компоненты, число поисков, gap_note-остатки.
- **После всех досье** — отдельным решением пользователя: пере-оценка impact для
  `improbable`/`discredited` с сегментацией по `failing_component_kind` (хартия §11),
  обновление синтеза и отчёта (framing-gap-распределение — полноценный результат).

## Фаза 2b — пере-оценка impact и обновление синтеза (выполнена 2026-07-05)

Вход: все 23 досье Фазы 2a со статусом `done` (81 обвинительное решение as-written:
improbable/discredited; плюс GC-18-058 со статусом out-of-scope → «условная посылка»).

- **Impact re-score.** Сабагент `impact-assessor` на каждую затронутую главу (24 главы,
  свежий контекст), вход — подготовленная выжимка ruling-записей + текст главы. Выход —
  дельта `impact/gc_chNN.rescore.delta.json`; оркестратор сливает её в
  `impact/gc_chNN.impact.json` НОВЫМ ключом `impact_adjudicated` через
  `scripts/merge_stage.py --key impact_adjudicated` (объекты Phase-1 `impact` не
  меняются), затем `scripts/validate.py`. Оценка ведётся против адъюдикационного
  решения, не против Phase-1-вердикта; в каждом объекте — `failing_component_kind`
  (fact/frame/motive — отдельные классы), `chain_inherited` +
  `independent_failure_point` (поправка 3 хартии), `framing_gap`.
  GC-18-058 несёт `category: "conditional_premise"` без severity/impact_score —
  downstream не считает его ни расхождением, ни оправданием.
- **Синтез (повторный прогон).** Один запуск сабагента `synthesizer` (`claude-fable-5`):
  обновлены `synthesis/aggregate.json`, `synthesis/clusters.md`,
  `synthesis/central_thesis_analysis.md` — распределения решений и framing_gap как
  полноценный результат, независимые точки провала vs унаследованные по цепочке,
  period_standard-теги (с различением «стандарт науки эпохи» / «стандарт традиции
  толкователей» по D04), распределение направлений (§7 хартии), категория «условная
  посылка», обновлённый анализ вычитания и машиночитаемые статусы 7 звеньев цепи
  тезиса в aggregate.json.

## Фаза 2c — итоговый сайт (следующая)

Интерактивный сайт на русском поверх полного слоя данных (Phase 1 + адъюдикация +
re-score + синтез). Заменяет `report/index.html`. Требования Фазы 3 (тепловая карта,
фильтруемая таблица, визуализация цепи тезиса — теперь по статусам звеньев из
aggregate.json, метрики) наследуются и расширяются: классы провалов
(fact/frame/motive) рендерятся по-разному, framing_gap и «условная посылка»
показываются как полноценные категории.
